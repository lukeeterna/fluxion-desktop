================================================================================
TEST SCENARIOS (10 TOTAL)
================================================================================

1. TestHappyPath
   - Complete booking from greeting to exit
   - Validates: full flow, state transitions, exit intent

2. TestBareName
   - Bare name from IDLE enters booking state
   - Validates: system enters booking on name alone

3. TestAmbiguousName
   - Name matching multiple clients triggers disambiguation
   - Validates: handles name collisions

4. TestNewClientRegistration
   - Unrecognized name triggers registration flow
   - Validates: new client pathway

5. TestGoodbye
   - Goodbye/exit at any point in conversation
   - Tests: from idle, mid-booking, multiple variants
   - Validates: exit intent recognized, should_exit=True

6. TestNameCorruption
   - Service name not confused with person name
   - Validates: "barba" ≠ "Barbieri"

7. TestFAQ
   - FAQ questions return correct information
   - Tests: price, hours
   - Validates: information retrieval

8. TestCancel
   - Cancel/modify operations mid-booking
   - Validates: cancellation pathway

9. TestOperatorEscalation
   - Operator request from idle and mid-booking
   - Validates: escalation intent recognized

10. TestEdgeCases
    - Empty string, very long input, punctuation, phone numbers
    - Special characters, mixed language
    - Validates: graceful handling, no crashes

Plus:
  - TestLatency: response times < 2 seconds
  - TestStateTransitions: logical FSM transitions
  - TestDifferentVerticals: salone, palestra, medical

================================================================================
RESPONSE STRUCTURE (what each turn returns)
================================================================================

{
  "success": true,                    # API call succeeded
  "response": "...",                  # Sara's response text
  "intent": "cortesia",               # Detected intent
  "fsm_state": "idle",                # Current FSM state
  "should_exit": false,               # Whether to exit conversation
  "should_escalate": false,           # Whether to escalate to operator
  "needs_disambiguation": false,      # Whether name disambiguation needed
  "layer": "l1_exact_match",          # Which RAG layer answered
  "session_id": "sess_12345",         # Session identifier
  "_latency_ms": 450.0                # Response time in milliseconds
}

Tests validate:
  - success == true (no server error)
  - response contains expected keywords
  - fsm_state is in expected state set
  - should_exit == true for goodbye
  - should_escalate == true for operator requests
  - _latency_ms < 2000

================================================================================
FSM STATES (valid values)
================================================================================

idle                    - No conversation
waiting_name            - Asking for client name
waiting_surname         - Asking for surname (disambiguation)
waiting_service         - Asking which service
waiting_date            - Asking for date
waiting_time            - Asking for time
confirming              - Asking for confirmation
completed               - Booking confirmed
cancelled               - Booking cancelled
disambiguating_name     - Resolving name collision
waiting_operator        - Escalated to operator
propose_registration    - New client registration
registering_phone       - Asking for phone during registration
registering_confirm     - Confirmation during registration
asking_close_confirmation - Asking before exit

================================================================================
ASSERTION PATTERNS
================================================================================

Pattern matching (recommended):
  assert_response_contains(response, "nome", "chi")
  # Passes if response contains "nome" OR "chi" (case-insensitive)

With must-not-contain:
  assert_response_contains(response, "euro", must_not_contain=["non"])
  # Passes if response contains "euro" AND does NOT contain "non"

State validation:
  assert t1["fsm_state"] in ("waiting_name", "waiting_service")
  # Passes if state is one of the expected values

Latency validation:
  assert t1["_latency_ms"] < 2000
  # Passes if response time under 2 seconds

Success check:
  assert t1["success"] is True
  assert t1.get("_connection_failed") is not True

  → Verify database schema hasn't changed

Response doesn't contain expected keywords
  → May be code deployed on iMac is old
  → Restart pipeline with latest code
  → Check orchestrator logs for NLU/intent issues

Test passes locally but fails on remote
  → Ensure same code deployed on both machines
  → Check PIPELINE_URL environment variable is correct
